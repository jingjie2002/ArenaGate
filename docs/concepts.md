# ArenaGate 和 CoreRank 到底有什么用

这份文档只回答一个问题：这两个项目如果放到游戏里，到底管什么。

## 先用现实比喻理解

假设你去线下打比赛：

- 门口前台：确认你是谁，给你登记，让你进入大厅。
- 排队系统：根据段位或水平帮你找对手。
- 桌号分配：告诉你去几号桌。
- 裁判和比赛桌：真正执行比赛规则。

对应到项目：

| 现实角色 | 游戏服务 | 项目 |
|---|---|---|
| 门口前台 | 网关 / 接入层 | ArenaGate |
| 排队系统 | 匹配中台 | CoreRank |
| 桌号分配 | 房间资源分配 | CoreRank |
| 比赛桌 | 房间服 / 战斗服 | CoreRank 里只有最小 roomserver 原型 |

## CoreRank 控制什么

CoreRank 控制“匹配和排名”。

它回答这些问题：

- 玩家当前排行榜分数是多少？
- 玩家要不要进入匹配队列？
- 哪两个玩家适合被匹配到一起？
- 匹配成功后生成哪个 `match_id`？
- 这局应该分配哪个 `room_id`？
- 客户端后续应该连接哪个 `server_addr`？

它不负责：

- 玩家 TCP/WebSocket 是否在线。
- 玩家断线怎么办。
- 玩家发来的每条消息是否合法。
- 角色移动、攻击、技能、伤害。

## ArenaGate 控制什么

ArenaGate 控制“玩家接入和在线连接”。

它回答这些问题：

- 玩家能不能连接进来？
- 这条 WebSocket 连接属于哪个 `player_id`？
- 这个玩家有没有先鉴权？
- 玩家发来的消息是否太大、太快、格式不对？
- 玩家发起匹配时，应该帮他调用 CoreRank 哪个接口？
- CoreRank 返回匹配结果后，怎么主动推给玩家？

它不负责：

- 匹配算法。
- 排行榜存储。
- 房间内玩家 ready 状态。
- 战斗内角色同步。
- 真实账号密码登录。

## 在游戏里的典型流程

```text
1. 玩家打开游戏。
2. 客户端连接 ArenaGate 的 WebSocket。
3. 客户端发送 auth，ArenaGate 记录这条连接属于 p1。
4. 玩家点击“开始匹配”。
5. 客户端发送 enqueue_match。
6. ArenaGate 调用 CoreRank 创建匹配票据。
7. CoreRank 等待另一个合适玩家。
8. 匹配成功后，CoreRank 生成 match_id / room_id / server_addr。
9. ArenaGate 查询到匹配成功。
10. ArenaGate 给玩家推送 match_found。
11. 客户端根据 server_addr 去连接房间服或战斗服。
```

## 这两个项目为什么适合放在简历里

CoreRank 展示的是：

- Redis 数据结构。
- 匹配生命周期。
- 排行榜。
- 房间资源分配。
- 后端中台接口。
- gRPC / RESTful / metrics。

ArenaGate 展示的是：

- Go 网络编程。
- WebSocket 长连接。
- session 生命周期。
- 心跳和超时。
- 限流和协议校验。
- 服务间调用。
- 给在线玩家主动推送结果。

这两个项目合在一起，能讲出一条完整链路：

```text
玩家上线 -> 连接网关 -> 发起匹配 -> 匹配中台配对 -> 分配房间 -> 网关推送结果 -> 玩家连接房间服
```

现在还没有做完整战斗服，所以不能说“我做了完整游戏后端”。更准确的说法是：

```text
我做了游戏匹配中台和玩家长连接接入网关，并用最小房间服原型打通了匹配到入房的链路。
```
